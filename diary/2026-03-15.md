- frontend/amplify/backend ファイルになっていることがベストなパスと名前なのか確認した
    - 結果、名前もパスもこれでよかった
    - この名前で amplify 下に置かなければいけないファイルであるため
- Amplify 用の IAM ロール作成 AmplifyBackendDeployRole
    - cdk bootstrap のようなものでアカウントレベルのリソース変更でデストロイ不要
    - よって CDK での処理ではなく手動でのマネジメントコンソール設定が推奨
    - 既に Amplify 用のカスタマー管理ロール CloudWatch Logs が作成されていたがそれとは別に作成する
- 作成したロールはデプロイされた amplify アプリにアタッチする
    - なので CF からアプリ自体は削除しない、これ自体は働いていない
    - backend リソースは別に CF にあるので、そちらのみを毎回 destroy すれば良い
    - アプリ自体を消してもいいが、毎回ロールのアタッチが必要になるため
    - なお sanbox 開発環境では不要 ローカルの接続プロファイルを使用するため
- 環境の使い分け
    - 有効化された Amplify アプリ GitHub CICD が必須で sandbox とは別
    - 上しか Amplify アプリとしてリストされない
    - sandbox: cfn でデプロイされている Amplify アプリではない
    - sandbox で開発 -> git push -> Amplify でデリバーされる
    - 両方有効化するのが正しい
- pipelined App の立ち上げ方法
    - npm run amplify:pipeline-deploy は打たない
    - mng-console で手動操作で立ち上げ GitHub フックで自動デプロイされる
    - Amplify > Hosting > Build Settings > amplify.yml は初回デプロイ成功までは手動でアップデートする
    - pipeline 化されれば、自動で変更追従する
